# ADR-002: 상태 관리 3종 분리 — Zustand · Jotai · React Query

- 상태: 채택 (Accepted)
- 일자: 2026-04
- 작성자: 개인 프로젝트

## 컨텍스트

이 앱이 다루는 상태는 **수명·소유자·동기화 정책이 명확히 다르다**.

| 상태 종류           | 예                                          | 수명             | 동기화 정책                            |
| ------------------- | ------------------------------------------- | ---------------- | -------------------------------------- |
| 서버 상태           | habits, habit_logs, coach_events 통계       | 다른 기기와 공유 | 캐시 무효화 · 재페치 · 낙관적 업데이트 |
| 전역 UI 상태        | 로그인 유저, 선택 날짜, 다크 모드 테마      | 앱 세션          | 단일 진실 + 일부 persist               |
| 트리 안의 원자 상태 | 편집 모드, 드래그 임시 순서, 카드 펼침 여부 | 컴포넌트 트리    | 휘발                                   |

이를 한 라이브러리(Redux 또는 Zustand 단일)로 모두 처리할 수도 있다. 가능은 하지만, 그렇게 했을 때 발생하는 비용이 분명히 존재한다.

## 결정

세 라이브러리로 책임을 **물리적으로** 분리한다.

- **React Query** — 서버 상태 전부 담당. habit / log / stat의 **모든 캐시·무효화·낙관적 업데이트(optimistic update)** 가 한 곳에 모이도록 한다.
- **Zustand** — 전역 UI 상태 담당. 로그인 유저, 선택 날짜, 다크 모드 테마, 코치 카드 쿨다운(persist). 모듈 단위 store로 분리.
- **Jotai** — 트리 안의 원자 상태 담당. 편집 모드 토글, 드래그 임시 순서 등 컴포넌트 트리에 묶인 짧은 수명 상태.

## 결과 / 트레이드오프

1주간의 안정화 사이클을 거치며 이 분리가 실제로 **얼마나 도움이 되는지** 한 번 검증되었다.

- **B4 (캐시 무효화 누락)** — 체크해도 `/stats`의 일별 트렌드에서 오늘 막대가 0%에 박혀 있던 결함. 원인은 `useToggleHabitLog.onSettled`가 byDate / byMonth만 무효화하고 **weekly 파생 키**를 빠뜨린 것. 수정은 **React Query 한 곳**에서 키 한 줄 추가로 끝났다.
  - 만약 서버 상태를 Zustand로 직접 관리했다면, **어떤 셀렉터가 어느 mutation 이후에 stale 상태가 되는지**를 매번 직접 추적해야 했을 것이다. React Query에 위임함으로써 "이 mutation 이후에 어떤 query를 invalidate할지"가 **한 곳에 코드로 표현**된다.
- **B5 (사용자 경계 누수)** — `signOut`이 IndexedDB만 비우는 데 그치면, `useCoachStore` persist의 쿨다운 타임스탬프가 다음 사용자에게 그대로 인계된다. **전역 UI 상태가 어디에 있는지를 한 라이브러리로 한정해둔** 덕분에, `signOut`에서 비워야 할 대상이 "Zustand persist + IndexedDB 두 곳"으로 명확했다. 만약 서버 상태와 UI 상태가 한 store에 섞여 있었다면, 어디까지 비울지에 대한 판단이 훨씬 복잡해졌을 것이다.
- **Jotai의 트레이드오프** — 사실 Jotai 자리는 React의 `useState` + Context로도 충분했다. Jotai를 굳이 쓰는 실익은 (a) 같은 atom을 다른 트리 위치에서 재구독할 수 있다는 점, (b) Zustand store가 **전역**인 데 비해 atom은 **지역적**이라는 의도 표명이다. 1인 코드베이스에서는 다소 **과한 선택**에 가깝다.

### 트레이드오프

- 학습 비용 — 새로 합류하는 개발자는 3종 라이브러리와 각각의 사용 패턴을 익혀야 한다. **팀 프로젝트**라면 정당화하기가 더 어려운 선택이다.
- 결정 비용 — 새로운 상태가 생길 때마다 "어디에 두지?"를 매번 판단해야 한다. 기준이 분명하지 않으면 표류하기 쉽다. 이 ADR의 분류표가 그 기준 역할을 한다.

## 대안 검토

- **Zustand 단일** — 가능하다. 다만 서버 상태의 캐시·재페치·낙관적 업데이트를 직접 구현해야 한다. React Query가 **이미 잘 풀어둔 문제**를 다시 푸는 비용이 크다.
- **Redux Toolkit + RTK Query** — RTK Query가 React Query 자리를 대체할 수 있고, **단일 store**라는 장점도 있다. Redux의 보일러플레이트 비용을 어떻게 평가하느냐의 문제. 이 프로젝트에서는 **Zustand의 가벼움**을 우선했다.
- **React `useState` + Context** — 트리 안 원자 상태에는 충분하다. 다만 전역 UI 상태(테마, 로그인 유저)에 Context만 사용하면 **리렌더링 폭탄**이 쉽게 발생한다. Zustand는 선택적 구독으로 그 비용을 차단한다.
